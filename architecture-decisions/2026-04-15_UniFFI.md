# 2026-04 FFI

## Decision

We are using [uniffi](https://github.com/mozilla/uniffi-rs) to generate language bindings for `go` and `wasm`.
We use a separate crate `csaf-ffi` to expose the functionality.

## Context

We want to provide the functionality of this library to other languages without the overhead of rewriting everything in those languages natively. Rust is used a base because it should provide the best performance in combination with developer experience.

## Alternatives

- We considered to merge `csaf-ffi` into `csaf-rs`, but this would bring a few negative consequences:
    - Uniffi would pollute downstream consumers of our library, feature gating would be complex
    - Type conflicts would arise (rlib vs cdylib/staticlib)
    - Rust edition would be limited to `2021` (by uniffi)
- [BoltFFI](https://www.boltffi.dev/docs/overview) doesn't have support for GO yet. Maybe update to this in the future as it claims to be much faster than uniffi.

## Consequences

- We have to manually update `csaf-ffi` when our traits/types change in order to expose the correct interface to the other languages
- more build steps or pipelines are needed as the different language targets handle bundling, deployment and consumption differently 

## Confidence Level

Good, works for now and uniffi has support for several other languages.
